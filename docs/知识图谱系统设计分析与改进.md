# 知识图谱系统设计分析与改进

## 一、现状架构概览

### 1.1 数据模型（Neo4j）

| 节点类型 | 主键属性 | 用途 |
|---------|---------|------|
| Character | characterId | 人物 |
| Foreshadowing | foreshadowingId | 伏笔（未解决/已解决） |
| PlotThread | threadId | 剧情线程，与伏笔关联 |
| Event | id | 事件 |
| Location | id | 地点 |
| Faction | id | 势力 |
| Artifact | id | 法宝 |
| Technique | id | 功法 |

### 1.2 写入时机与来源

| 阶段 | 写入内容 | 来源 |
|-----|---------|------|
| SeedExecuteNode | 主角 Character | protagonistSetting 解析 |
| ChapterExecuteNode | keyCharacters、Foreshadowing、Event、PlotThread、Character→Event、Character→PlotThread、PlotThread→Foreshadowing | ChapterOutline |
| SceneExecuteNode | Scene.characters、Event、Location、Character→Event、Character→Location、PlotThread→Event | Scene + Outline |
| ExtractedEntitySyncService | 人物/地点/势力/法宝/功法/事件/关系/伏笔 | InfoExtractionAgent |

### 1.3 读取与消费

- **getGraph**：前端可视化，含启发式连线（StoryHub、Character→Foreshadowing MENTIONS）
- **buildStoryKnowledge**：SceneGenerationAgent 用，拉取 characterFacts、worldFacts、activePlotThreads
- **listUnresolvedForeshadowing**：PlotTracker、EndingAgent 用
- **listActivePlotThreads**：PlotTracker、EndingAgent 用

## 二、问题与改进方向

### 2.1 伏笔与剧情线程冗余

**问题**：同一伏笔会创建 Foreshadowing 与 PlotThread 两套节点，内容重复；`listUnresolvedForeshadowing` 与 `listActivePlotThreads` 各自查询，可能不一致。

**改进**：
- 保持双节点（Foreshadowing 为"待回收"语义，PlotThread 为"剧情线"语义），通过 PlotThread→Foreshadowing(REFERS_TO) 显式关联
- 伏笔回收时同时更新 Foreshadowing.resolved 与 PlotThread.status，保证一致性

### 2.2 伏笔回收机制缺失

**问题**：有 `resolved` 字段，但流程中无主动回收入口；结局判定后未批量标记。

**改进**：
- 新增 `resolveForeshadowing(foreshadowingId)`、`resolveForeshadowings(novelId, ids)`
- 在结局生成或章节明确回收时调用；EndingAgent 判定结束时可选批量回收（可配置）

### 2.3 人物类型泛化

**问题**：ExtractedEntitySyncService 将所有抽取人物标为「配角」，无法区分主角/重要配角。

**改进**：
- 抽取时若 outline.keyCharacters 含主角名，对应 Character 保持「主角」；其余按关系推断（如 relations 中 MENTOR_OF 的 subject 可标为重要角色）

### 2.4 buildStoryKnowledge 人物范围偏窄

**问题**：仅用 `outline.keyCharacters` 查 KG，若 outline 只有主角，则只拉主角的 facts。

**改进**：
- 合并 `keyCharacters` 与 `extractedEntities.characters`（或 scene.characters）作为查询列表，扩大上下文

### 2.5 同步逻辑分散

**问题**：ChapterExecuteNode、SceneExecuteNode、ExtractedEntitySyncService 多处写 KG，职责重叠、难以复用。

**改进**：
- 抽取 `KgSyncFacade` 或统一写入规范，各节点通过 Facade 调用，便于测试与演进（中长期）

### 2.6 关系幂等与去重

**现状**：createCharacter、createRelationship、upsertEntity 均用 MERGE，已具备幂等性。✓

### 2.7 跨类型关系利用不足

**问题**：Character→Faction、Character→Artifact 等关系已写入，但 buildStoryKnowledge 未充分利用。

**改进**：
- 在 loadCharacterFacts 中扩展，拉取人物与 Faction/Artifact 的关系，拼接为 worldFacts 或 characterFacts

## 三、已实施改进

1. **人物抽取加强**：ChapterOutlineAgent 基于梗概提取关键人物，InfoExtractionAgent 抽取正文中所有人物
2. **启发式连线**：StoryHub→全部节点、Character→Foreshadowing(MENTIONS)
3. **getGraph 全类型边**：查询所有关系类型，包含 Event、Location、PlotThread 等
4. **PlotThread→Foreshadowing**：建立 REFERS_TO 边，伏笔与剧情线关联

---

## 四、已实施代码改进（本次）

1. **伏笔回收机制**
   - 新增 `resolveForeshadowing(id)`、`resolveForeshadowings(novelId, ids)`
   - 新增 `listUnresolvedForeshadowingIds(novelId)` 供批量回收
   - ValidationExecuteNode：结局判定通过时自动批量回收未解决伏笔
   - 回收时同步更新 PlotThread.status = 'RESOLVED'，保证与 Foreshadowing.resolved 一致

2. **人物类型推断（ExtractedEntitySyncService）**
   - 结合 outline.keyCharacters 首项推断主角（支持「主角'林衍'」格式）
   - 与 protagonist 同名的抽取人物标为「主角」，其余为「配角」

3. **updateForeshadowing 行为**
   - content 为 null 时不再覆盖原有 content，仅更新 resolved 等字段

## 五、后续建议

2. 考虑 Foreshadowing 与 PlotThread 的长期统一或显式映射文档
3. 抽取 KgSyncFacade 统一写入入口，便于测试与演进
