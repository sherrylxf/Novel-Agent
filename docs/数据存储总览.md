# Novel Agent 数据存储总览

本文档用于帮助开发者快速理解 **Novel Agent** 项目中“所有数据存储”的组成、用途、数据流向与关键配置。

> 一句话总结：  
> - **MySQL**：业务/生成结果的主存储（Seed/Plan/Volume/Chapter/Scene 等）。  
> - **PostgreSQL + pgvector**：RAG 向量检索存储（相似文本/风格检索）。  
> - **Neo4j**：知识图谱（人物/关系/伏笔/规则校验）。  
> - **本地 TXT 文件**：每章正文导出，方便离线阅读/归档/二次加工。

---

## 1. 存储系统与职责边界

### 1.1 MySQL（业务数据主库）
- **用途**：保存小说生成过程中的结构化结果与正文内容，是“最终产物”的权威来源（source of truth）。
- **典型数据**：
  - `novel_seed`：小说种子（题材/核心冲突/世界观/主角设定等）
  - `novel_plan`：整体规划（总卷数/每卷章节数/整体大纲）
  - `volume_plan`：卷规划（卷标题/卷主题/章节数）
  - `chapter`：章节（梗概 + 正文）
  - `scene`：场景（正文场景、人物、地点等）
- **配置位置**：`ai-agent-station-study-app/src/main/resources/application-dev.yml`
  - `spring.datasource.mysql.*`

### 1.2 PostgreSQL + pgvector（向量库 / RAG）
- **用途**：为 RAG 提供向量检索能力（按相似度检索历史片段/风格片段）。
- **典型数据**：
  - 向量表（embedding + metadata + content）
- **配置位置**：`ai-agent-station-study-app/src/main/resources/application-dev.yml`
  - `spring.datasource.pgvector.*`
- **相关脚本**：
  - `docs/dev-ops/pgvector/create-table.sql`
  - `docs/dev-ops/pgvector/init.sql`

### 1.3 Neo4j（知识图谱 / KG）
- **用途**：存人物、关系、伏笔等图谱信息；并可支持规则校验（Consistency/KGRuleValidator）。
- **典型数据**：
  - `(:Character {characterId, name, type, personality, background, abilities})`
  - `(:Foreshadowing {foreshadowingId, content, ...})`
  - 关系如：`(Character)-[:FRIEND|ENEMY|FAMILY|...]->(Character)`
- **配置位置**：`ai-agent-station-study-app/src/main/resources/application-dev.yml`
  - `spring.neo4j.uri`
  - `spring.neo4j.authentication.username`
  - `spring.neo4j.authentication.password`
- **实现类**：`ai-agent-station-study-infrastructure/src/main/java/cn/bugstack/novel/infrastructure/adapter/kg/Neo4jKnowledgeGraphService.java`

### 1.4 本地 TXT（章节导出）
- **用途**：把“每章正文”导出为 txt，便于开发者/运营/写作人员离线查看与归档。
- **默认目录**：`./output/chapters`（相对应用启动目录）
- **配置键**：`novel.export.base-dir`
- **实现类**：`ai-agent-station-study-infrastructure/src/main/java/cn/bugstack/novel/infrastructure/adapter/export/LocalChapterFileStore.java`

## 2. MySQL 表结构（核心表）

MySQL 建表脚本参考：
- `docs/dev-ops/mysql/sql/novel-agent.sql`

> 下述字段以脚本为准，这里按“业务理解”做速览。

### 2.1 `novel_seed`（小说种子）
- **关键字段**：
  - `seed_id`（唯一）
  - `novel_id`
  - `title`
  - `genre`
  - `core_conflict`、`world_setting`、`protagonist_setting`
  - `target_word_count`

### 2.2 `novel_plan`（整体规划）
- **关键字段**：
  - `plan_id`（唯一）
  - `novel_id`
  - `total_volumes`
  - `chapters_per_volume`
  - `overall_outline`

### 2.3 `volume_plan`（卷规划）
- **关键字段**：
  - `volume_id`（唯一）
  - `novel_id`
  - `volume_number`（卷序号）
  - `volume_title`
  - `volume_theme`
  - `chapter_count`

### 2.4 `chapter`（章节：梗概 + 正文）
- **关键字段**：
  - `chapter_id`（唯一）
  - `novel_id`
  - `volume_number`
  - `chapter_number`
  - `title`
  - `outline`（梗概）
  - `content`（正文，`longtext`）
  - `word_count`
  - `status`（0 草稿 / 1 完成）

### 2.5 `scene`（场景正文）
- **关键字段**：
  - `scene_id`（唯一）
  - `chapter_id`
  - `scene_number`
  - `scene_title`
  - `scene_type`
  - `content`（正文，`longtext`）
  - `word_count`
  - `characters`（JSON 数组字符串）
  - `location`

## 3. 生成链路的数据落库点（非常重要）

项目使用责任链节点执行（分步确认模式时，每次只执行一个节点）。

执行节点链路（概念上）：
`RootExecuteNode → SeedExecuteNode → PlanExecuteNode → VolumeExecuteNode → ChapterExecuteNode → SceneExecuteNode → ValidationExecuteNode`

### 3.1 自动落库策略
- **Seed 生成后**自动写 `novel_seed`
  - 位置：`SeedExecuteNode`
- **Plan 生成后**自动写 `novel_plan`
  - 位置：`PlanExecuteNode`
- **Volume 生成后**自动写 `volume_plan`
  - 位置：`VolumeExecuteNode`
- **Scene 生成后**自动写 `chapter`（梗概+正文）与 `scene`
  - 位置：`SceneExecuteNode`

> 设计原则：  
> - “文件导出 / 图谱写入”失败 **不阻塞主流程**（只告警）。  
> - 对同一 `novel_id + volume_number + chapter_number` 的重复生成，走“存在则更新”，避免唯一键冲突和重复插入。

## 4. PgVector（向量库）数据流

典型流向：
- **检索（读）**：生成场景正文时（`SceneGenerationAgent`）
  - 通过 `IRAGService.search(query, "zh", topK, novelId)` 从向量库检索**仅限本小说**的相似场景作为参考上下文，避免跨小说干扰。
- **写入（写）**：场景落库后（`SceneExecuteNode`）
  - 在 `persistChapterAndScene` 成功后，将 `Scene.content` 调用 `IRAGService.addDocument(content, "zh", metadata)` 写入向量库。metadata 包含 novelId、卷/章/场景信息，以及 **characters**（参与人物，逗号分隔）、**location**（地点），便于后续按人物或地点筛选与展示。失败仅打 warn，不阻塞主流程。

如何确认 RAG 已写入：查看日志关键字 `RAG存储: 场景已写入向量库` 或 `RAG存储: 文档已写入向量库(PgVector)`。若为占位 RAG 或 VectorStore 未配置，则不会出现上述日志。

注意：
- 向量库连接可选；未配置/不可用时，需要确保上层有降级策略（项目中已有降级处理思路）。

## 5. Neo4j（知识图谱）数据流

### 5.1 KG 写入/查询
- KG 服务接口：`cn.bugstack.novel.domain.service.kg.IKnowledgeGraphService`
- Neo4j 实现：`Neo4jKnowledgeGraphService`
  - 创建人物：`createCharacter(...)`
  - 创建关系：`createRelationship(...)`
  - 查询人物/关系：`queryCharacter(...)` / `queryRelationships(...)`
  - 伏笔：`createForeshadowing(...)`（建议 properties 含 `novelId`、`chapterId`、`resolved`）
  - **未解决伏笔列表**：`listUnresolvedForeshadowing(novelId)`（供 Plot Tracker / EndingAgent 使用）
  - 规则校验：`validateRule(...)`

### 5.2 KG 写入时机（人物、关系、伏笔）
- **Seed 生成后**（`SeedExecuteNode`）：从 `NovelSeed.protagonistSetting` 解析主角名，调用 `createCharacter` 写入主角节点（type=主角，background=设定全文）。失败仅打 warn，不阻塞。
- **章节梗概生成后**（`ChapterExecuteNode`）：  
  - 若 `ChapterOutline.foreshadowing` 非空，逐条调用 `createForeshadowing(..., { novelId, chapterId, resolved: false })` 写入伏笔。  
  - 将 `ChapterOutline.keyCharacters` 逐个 `createCharacter`（同一 novelId 下人物 ID 为 `novelId_c_<规范化名>`，类型默认「配角」）。  
  - 若本章关键人物 ≥2，两两建立 `RELATED_TO` 关系边（属性含 chapterId、chapterNumber），丰富人物关系图。
- **场景生成后**（`SceneExecuteNode`）：将 `Scene.characters` 中的人物同步到 KG（不存在则创建），保证「出现即存在」。

如何确认 KG 已写入：查看日志关键字 `KG存储: 伏笔已写入知识图谱`、`KG存储: 主角已写入知识图谱`、`KG存储: 本章关键人物已同步到知识图谱`、`KG存储: 场景人物已同步到知识图谱`。若为占位 KG 或 Neo4j 未配置，则不会写入。

### 5.3 校验阶段与多章收束
- `ValidationExecuteNode` 会调用：
  - `ConsistencyGuardAgent`
  - `KGRuleValidatorAgent`
  - **Plot Tracker**（`IPlotTrackerService.buildProgress`）从上下文 + KG 未解决伏笔构建 `StoryProgress`
  - **EndingAgent** 根据剧情摘要、章节进度、未解决伏笔判定 `shouldEndStory`；若为 true 则流程结束，否则进入下一章/下一卷（多章循环）。

注意：
- Neo4j 是可选能力；如果没有 `Driver` Bean，KG 服务可能走占位/不生效实现，校验可降级为“继续流程但提示告警”。

## 6. TXT 导出（章节正文）

### 6.1 导出时机
- 在 `SceneExecuteNode` 生成出 `Scene.content` 后导出。

### 6.2 路径与命名
- 输出根目录：`novel.export.base-dir`（默认 `./output/chapters`）
- 文件组织：
  - `<base>/<novelId>/volume-01/chapter-001-章节标题.txt`
- 文件名安全：
  - 会清洗 Windows 不允许字符 `<>:"/\\|?*` 与控制字符
  - 标题过长会截断

## 7. 开发者自检清单（快速排障）

- **MySQL 落库不生效**
  - 检查 `application-dev.yml` 的 `spring.datasource.mysql.*`
  - 检查数据库是否执行了 `docs/dev-ops/mysql/sql/novel-agent.sql`
  - 观察控制台是否有 MyBatis `INSERT/UPDATE` 日志

- **pgvector 检索报错**
  - 检查 `spring.datasource.pgvector.*`
  - 检查是否已执行 pgvector 初始化脚本

- **Neo4j 连接失败**
  - 检查 `spring.neo4j.uri`（本地默认 `bolt://localhost:17687`）
  - 检查账号密码
  - 不使用 KG 时可保持可选（不影响主流程）

- **TXT 没有生成**
  - 检查 `novel.export.base-dir` 指向目录是否有写权限
  - 检查是否走到了 `SceneExecuteNode`（场景生成完成后才导出）
